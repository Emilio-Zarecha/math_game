Chapter 1

Graphs, Vertices, and Edges

A map of flights between cities, a social network of friendships, and a diagram of which web pages link to which — all three are the same mathematical object wearing different clothes. Whenever a problem is really about a set of things and the pairwise relationships between them, a graph strips away the surface detail and leaves only the structure that matters: who is connected to whom.

Diagram — a small undirected graph G

Formally, a graph $G = (V, E)$ consists of a set of vertices $V$ (also called nodes) and a set of edges $E$, where each edge connects a pair of vertices. In the diagram above, $V = \{A,B,C,D,E,F\}$ and each line is an edge — for example $\{A,B\}$ and the diagonal $\{A,D\}$. An edge with no direction means the relationship runs both ways: "A is friends with B" implies "B is friends with A." When direction matters instead — "A follows B" does not imply "B follows A" — the edges carry arrows and the graph is called directed (a digraph).

The degree of a vertex is the number of edges touching it. In graph G, vertex A touches edges to B, F, and D, so $\deg(A) = 3$; vertex B touches only A and C, so $\deg(B) = 2$. Every edge contributes exactly 2 to the total degree count (one at each endpoint), which gives a fact that holds for every graph, no matter how large: the sum of all degrees is always even.

Handshake Theorem

$\displaystyle\sum_{v \in V} \deg(v) = 2|E|$
The sum of vertex degrees always equals twice the number of edges — so the sum of degrees is always even.

Example — Verifying the Handshake Theorem on G
$\deg(A){=}3,\ \deg(B){=}2,\ \deg(C){=}2,\ \deg(D){=}3,\ \deg(E){=}2,\ \deg(F){=}2$ Count edges touching each vertex
$3+2+2+3+2+2 = 14$ Sum of all degrees
$14 = 2 \times 7$, and $G$ has exactly 7 edges Confirms the theorem: sum of degrees = 2|E|
⬡ AI Connection

A neural network's forward pass is itself a graph: each operation (matrix multiply, add bias, apply activation) is a vertex, and each edge carries a tensor from one operation to the next. Frameworks like PyTorch and TensorFlow literally build this graph in memory before running anything — "vertices and edges" is not a metaphor here, it's the data structure the framework executes.

Chapter 2

Graph Representations

Drawing a graph works for six vertices on a page. It breaks down completely for a social network with a billion users. To compute anything about a graph — is it connected? what's the shortest path? — a computer needs the graph encoded as data, not a picture. There are two standard encodings, and the right choice depends entirely on whether the graph is dense (many edges) or sparse (few edges relative to the number of possible pairs).

Diagram — adjacency matrix for G (from Chapter 1)

An adjacency matrix is an $n \times n$ grid where entry $(i,j)$ is 1 if an edge connects vertex $i$ and vertex $j$, and 0 otherwise. For an undirected graph the matrix is always symmetric, since an edge $\{i,j\}$ makes both $(i,j)$ and $(j,i)$ equal to 1. Checking "is A connected to D?" is a single lookup — $O(1)$ — which makes the matrix ideal for dense graphs or when adjacency queries dominate. The cost is space: an $n$-vertex graph needs $n^2$ cells even if it only has a handful of edges.

An adjacency list instead stores, for each vertex, just the list of its neighbors. Vertex A's list would be $[B, D, F]$. This uses $O(|V| + |E|)$ space total — proportional to the actual number of edges, not the square of the vertex count — which is why almost every real-world graph library (road networks, social graphs, web link graphs) defaults to adjacency lists: those graphs are sparse, with each vertex connected to only a tiny fraction of all other vertices.

⬡ AI Connection

Graph Neural Networks consume exactly this adjacency information as their input: a node-feature matrix plus an adjacency structure (often a sparse adjacency list, since real graphs like citation networks or molecules are sparse). A GNN layer computes each node's new representation by aggregating its neighbors' features — reading straight off the same adjacency list you'd use for any other graph algorithm.

Chapter 3

Paths, Walks, and Connectivity

"Can I get from A to C?" is the single most common question asked of a graph — it's a route planner, a network reachability check, and a social-graph "degree of separation" query all at once. Answering it precisely requires being careful about the difference between a walk that might repeat vertices and a path that visits each vertex at most once.

Diagram — a path in G, and a disconnected vertex

A walk is any sequence of vertices where consecutive vertices are joined by an edge; it may reuse vertices and edges freely. A path is a walk that never repeats a vertex. A cycle is a path that returns to its starting vertex (e.g. A→B→C→D→A in graph G). Two vertices are connected if some path exists between them, and a graph is connected overall if every pair of vertices is connected. The isolated vertex Z above has no edges at all, so it forms its own connected component — the graph as a whole is disconnected.

Connectivity questions scale up in important ways. In a road network, a bridge (or cut edge) is an edge whose removal disconnects the graph — a single point of failure. A cut vertex plays the same role for a vertex. Both concepts matter enormously for network reliability: internet backbone design and power grid topology are both, at their core, exercises in avoiding cut edges and cut vertices.

Example — Is graph G connected?
Start at A, list every vertex reachable by some path Reachability defines connectivity
A→B, A→F, A→D, B→C, D→C, D→E Following edges from A outward
Reachable set = $\{A,B,C,D,E,F\}$ = all of $V$ Every vertex reached — G is connected
⬡ AI Connection

Backpropagation is a walk in reverse: to compute how the loss depends on an early-layer weight, the chain rule multiplies gradients along every path from the loss node back to that weight. If a weight's node has no path to the loss node — the analogue of a disconnected component — its gradient is exactly zero and it never updates during training.

Chapter 4

Graph Traversal: BFS and DFS

Knowing a graph is connected is one thing; systematically visiting every vertex is another, and it turns out there are exactly two fundamentally different disciplined ways to do it. Both show up constantly — in maze solvers, in web crawlers, in game-tree search — and the choice between them changes both the order vertices are visited in and the memory the algorithm needs.

Diagram — BFS visit order on G starting from A

Breadth-First Search (BFS) visits vertices in waves by distance from the start: first the start vertex, then all its direct neighbors, then all vertices two edges away, and so on. It uses a queue (first-in, first-out) to remember which vertex to visit next. Because it explores level by level, BFS is the standard way to find the shortest path (fewest edges) in an unweighted graph — the first time it reaches a vertex is guaranteed to be via a shortest route.

Depth-First Search (DFS) instead commits to one direction and follows it as far as possible before backtracking, using a stack (last-in, first-out) — implemented explicitly or, more commonly, via recursion. Starting at A, DFS might go A→B→C→D→E→F, plunging deep before exploring alternatives. DFS uses less memory for wide graphs, and it's the natural tool for tasks like detecting cycles, finding connected components, and topological sorting (Chapter 7).

Example — DFS visit order on G starting from A
Visit A, push its unvisited neighbors: B, F, D DFS picks one neighbor and commits
A → B → C → D (D's other neighbor E still unvisited) → E → F Follow each branch to its end before backtracking
Order: A, B, C, D, E, F Compare to BFS order A, B, F, D, C, E above — same vertices, different order
⬡ AI Connection

Minimax search in game-playing AI (chess, Go) is depth-first traversal of the game tree — commit to a line of play, evaluate it fully, then backtrack to try alternatives. Message passing in a Graph Neural Network is closer to breadth-first: in each round every node simultaneously gathers information from its immediate neighbors, exactly like BFS's wave-by-distance expansion.

Chapter 5

Trees and Spanning Trees

If you want to connect every city with fiber-optic cable at minimum cost, you don't need every possible link — you need the cheapest set of links that still connects everything, with zero redundant connections. That minimal connecting structure is always a tree, and finding it is one of the oldest solved problems in graph theory.

Diagram — a spanning tree of G (green edges)

A tree is a connected graph with no cycles. This single constraint forces a precise relationship between vertices and edges: any tree with $n$ vertices has exactly $n-1$ edges — one fewer edge than vertex, since each new vertex attaches via exactly one edge without ever closing a loop. A spanning tree of a graph $G$ is a subgraph that is a tree and touches every vertex of $G$. The diagram shows one of G's many possible spanning trees: 5 edges connecting all 6 vertices, with the diagonal $\{A,D\}$ and edge $\{F,A\}$ excluded to avoid closing a cycle.

When edges have weights (costs), the Minimum Spanning Tree (MST) is the spanning tree with the smallest total edge weight. Two greedy algorithms both provably find it: Kruskal's algorithm sorts all edges by weight and adds each one unless it would create a cycle; Prim's algorithm grows a single tree outward, always adding the cheapest edge that connects a new vertex. Both work because of a deep fact called the cut property: the cheapest edge crossing any partition of the vertices is always part of some MST.

Example — Kruskal's algorithm, sketch
Sort edges by weight, ascending Greedy: always consider the cheapest remaining edge
Add an edge only if its two endpoints are in different components Adding an edge within the same component would create a cycle
Stop once $n-1$ edges have been added A tree spanning all $n$ vertices needs exactly $n-1$ edges
⬡ AI Connection

A decision tree — one of the most widely used classical ML models — is a rooted tree in exactly this graph-theoretic sense: no cycles, one path from root to any leaf. Random forests and gradient-boosted trees (XGBoost, LightGBM) are ensembles of many such trees, and hierarchical clustering builds a "dendrogram" that is literally a spanning tree of the data points ordered by similarity.

Chapter 6

Shortest Path Algorithms

BFS finds shortest paths when every edge counts equally. Real routes rarely do — a flight costs money, a road segment takes time, a network hop has latency. Once edges carry weights, shortest-path finding needs a smarter algorithm than "explore in waves," and the most famous one is due to Edsger Dijkstra.

Diagram — weighted G, shortest path A to E highlighted

Dijkstra's algorithm finds shortest paths from one source vertex to all others, provided every edge weight is non-negative. It keeps a running "best known distance" for every vertex (starting at $\infty$ except the source, which is 0), and repeatedly picks the unvisited vertex with the smallest known distance, then relaxes its outgoing edges — updating a neighbor's distance if going through the current vertex is cheaper than what was known before. Because it always expands the cheapest frontier vertex next, once a vertex is finalized its distance can never improve.

In the diagram, the direct edge $A$–$D$ costs 6, and the route $A\to B\to C\to D$ costs $4+3+2=9$ — both worse than they might look at a glance. For the destination $E$, the route $A \to F \to E$ costs only $1 + 2 = 3$, beating both $A \to D \to E$ ($6+5=11$) and the long way around through $B$ and $C$. When negative edge weights are possible, Dijkstra can give wrong answers, and the slower Bellman-Ford algorithm — which relaxes every edge repeatedly, $|V|-1$ times — is used instead, since it can also detect negative-weight cycles.

Example — comparing routes from A to E
$A \to F \to E$: weight $1 + 2 = 3$ Two hops through F
$A \to D \to E$: weight $6 + 5 = 11$ Two hops through D — much more expensive
Shortest path: $A \to F \to E$, total weight $3$ Dijkstra would finalize F (distance 1) before D (distance 6), then find E via F
⬡ AI Connection

A* search — Dijkstra plus a heuristic estimate of remaining distance — is the standard pathfinding algorithm in robotics and game AI navigation. In reinforcement learning, value iteration propagates value estimates backward through the state-transition graph in a way that's structurally close to Dijkstra's edge relaxation: repeatedly improve an estimate using a neighbor's already-improved estimate.

Chapter 7

Directed Acyclic Graphs and Topological Sort

Course prerequisites, build-system dependencies, and spreadsheet formula recalculation all share one requirement: some things must happen before others, and there must be no circular requirement ("A depends on B depends on A" is a contradiction, not a valid plan). This is precisely the structure of a Directed Acyclic Graph (DAG).

Diagram — a DAG with one valid topological order

A DAG is a directed graph containing no directed cycles — you can never follow arrows and return to where you started. A topological sort is an ordering of the vertices such that every edge points from an earlier vertex to a later one. In the diagram, $P, Q, R, S, T$ is one valid order (P before its dependents Q and R; both before S; S before T) — and $P, R, Q, S, T$ is equally valid, since Q and R don't depend on each other. A topological sort exists if and only if the graph is acyclic; DFS produces one directly, by outputting vertices in reverse order of when DFS finishes exploring them.

This is exactly what a build system or task scheduler computes before running anything: given a dependency DAG, find *an* order (not necessarily unique) that respects every dependency. If the dependency graph has a cycle, no valid order exists — which is precisely how tools like `make` or a package manager detect and report circular dependencies.

Example — course prerequisites as a DAG
Calculus I → Calculus II, Calculus I → Linear Algebra, Calculus II → Diff. Eq. Arrows mean "must be taken before"
Valid order: Calculus I, Linear Algebra, Calculus II, Diff. Eq. Every course appears after all its prerequisites
If Diff. Eq. also required Linear Algebra → Calculus I, that edge would create a cycle No valid order could exist — the schedule would be impossible
⬡ AI Connection

This is the single most important graph structure in deep learning. A neural network's computation graph is a DAG by construction — no operation can depend on its own output — and reverse-mode automatic differentiation (backpropagation) works by topologically sorting the graph, running forward in that order, then running backward in the reverse order to accumulate gradients. Bayesian networks are also DAGs, where edges encode direct probabilistic dependence rather than data flow.

Chapter 8

Graph Coloring and Bipartite Graphs

Assign exam time slots so no student has two exams at once. Assign radio frequencies so no two overlapping towers interfere. Color a map so no two adjacent countries share a color. All three are the same problem: assign labels ("colors") to vertices so that no edge connects two vertices with the same label, using as few colors as possible.

Diagram — a bipartite graph, 2-colored

The minimum number of colors needed is the chromatic number, $\chi(G)$. Finding it exactly is NP-hard in general — there's no known efficient algorithm that works on all graphs — but one special case is easy to recognize and solve: a graph is bipartite if its vertices split into two groups with every edge crossing between groups and none within a group. A graph is bipartite if and only if it contains no odd-length cycle, and every bipartite graph has $\chi(G) = 2$ — exactly the two colors shown above, since by definition no edge ever connects two same-side vertices.

Bipartite graphs also support matching: pairing up vertices across the two sides so that each vertex is used at most once. The classic application is assigning applicants to jobs, or students to dorm rooms, where each side has preferences — the Hungarian algorithm finds an optimal matching efficiently, in polynomial time, even though general graph coloring does not have such a guarantee.

Example — why an odd cycle can't be bipartite
Triangle A–B–C–A: try to 2-color it 3-cycle, odd length
A = blue, B = green (forced, since A–B is an edge) Adjacent vertices must differ
C must differ from both B (green) and A (blue) — no color left Odd cycles always fail 2-coloring; this is the bipartite test
⬡ AI Connection

Graph coloring's classical use in compilers — assigning a limited set of CPU registers to variables so that no two "live at the same time" variables share one — is register allocation, and it directly affects how efficiently a trained model's compiled inference code runs. Bipartite matching also shows up in attention-adjacent problems: assigning a fixed set of experts to tokens in mixture-of-experts models is a bipartite matching problem at each layer.

Chapter 9

Eulerian and Hamiltonian Paths

In 1736, the city of Königsberg had seven bridges, and its residents wondered whether a walk existed that crossed every bridge exactly once and returned to the start. Leonhard Euler proved it was impossible — and in doing so, invented graph theory as a field. The question "visit every edge exactly once" and the very different question "visit every vertex exactly once" turn out to have sharply different levels of difficulty.

Diagram — an Eulerian path on G, edges numbered by visit order

An Eulerian circuit traverses every edge exactly once and returns to its start; an Eulerian path does the same but may end elsewhere. Euler's theorem settles both completely: a connected graph has an Eulerian circuit if and only if every vertex has even degree, and has an Eulerian path (but not a circuit) if and only if exactly two vertices have odd degree — the path must start at one and end at the other. Graph G has exactly two odd-degree vertices, A and D ($\deg(A){=}\deg(D){=}3$ from Chapter 1), so it has an Eulerian path but no Eulerian circuit — the numbered path above, $A \to F \to E \to D \to C \to B \to A \to D$, uses each of the 7 edges exactly once.

A Hamiltonian path instead visits every vertex exactly once (edges may be skipped entirely). It sounds like a small variation on the Eulerian question, but the two problems live in completely different complexity classes: Euler's even-degree test decides Eulerian existence instantly, while no efficient test is known for Hamiltonian paths in general — determining whether one exists is NP-complete. The famous Traveling Salesman Problem is the Hamiltonian question with the extra twist of minimizing total edge weight.

⬡ AI Connection

Because Hamiltonian-path and TSP-style problems resist exact efficient solutions, they are a standard benchmark for AI optimization methods: reinforcement learning agents, neural combinatorial optimization models, and metaheuristics like ant colony optimization are all routinely trained or tuned by trying to beat classical heuristics on TSP instances — a problem simple enough to state but hard enough to stay relevant.

Chapter 10

Network Flow

A pipeline network moving oil from a source to a refinery, each pipe segment able to carry only so much per hour — how much oil can actually get through, end to end? This is the maximum flow problem, and its answer connects to something that sounds completely different: the cheapest way to disconnect the network entirely.

Diagram — a flow network, source S to sink T

Every edge in a flow network has a capacity — the maximum amount it can carry — and flow must satisfy conservation: at every vertex except the source $S$ and sink $T$, flow in equals flow out. In the diagram, up to 10 units can flow $S\to A$ and up to 8 can then flow $A \to T$, so the $S$–$A$–$T$ route is bottlenecked at $\min(10,8)=8$; the $S$–$B$–$T$ route is bottlenecked at $\min(5,7)=5$. Since the two routes share no edges, the maximum total flow from $S$ to $T$ is $8+5=13$.

The Max-Flow Min-Cut Theorem proves this is no coincidence: the maximum possible flow through a network always exactly equals the minimum total capacity of edges that would need to be cut to fully separate the source from the sink. Finding the bottleneck and finding the max flow are the same computation viewed two ways — the Ford-Fulkerson method finds max flow by repeatedly finding any path from $S$ to $T$ with spare capacity and pushing more flow along it, until no such path remains.

Example — reading the min cut off the diagram
Cut $\{A\to T, B\to T\}$: total capacity $8+7=15$ Separates T from everything else
Cut $\{S\to A, S\to B\}$: total capacity $10+5=15$ Separates S from everything else
Neither is minimal — the true min cut is $8+5=13$, matching max flow Cutting A→T (8) and S→B (5) also fully separates S from T, at lower cost
⬡ AI Connection

Flow-network thinking underlies resource-constrained scheduling problems that show up in distributed model training — routing gradient updates or data shards through a limited- bandwidth cluster network is a max-flow-style optimization. It also gives a useful mental model for attention: a fixed "budget" of attention weight flowing from each query token, distributed across key tokens, is loosely analogous to flow conservation at a vertex.

Chapter 11

Graphs in Modern AI

Every chapter so far has included a short AI sidebar — this chapter is where those threads tie together. Nearly every major modern AI architecture is, underneath its name, a particular way of computing over a graph: the only real questions are what the vertices represent, what the edges represent, and what computation happens at each vertex.

Diagram — a small feedforward network as a layered computation graph

A Graph Neural Network (GNN) generalizes the feedforward picture above to arbitrary graph-structured data — molecules, social networks, road networks — where the "layers" aren't fixed by architecture but by the input graph's own edges. Each GNN layer performs message passing: every vertex gathers (aggregates) information from its immediate neighbors, exactly the BFS one-hop expansion from Chapter 4, and updates its own representation. Stacking $k$ layers lets information travel $k$ hops — a vertex's final representation depends on everything within distance $k$ in the graph.

A Transformer — the architecture behind large language models — can be read as a GNN operating on a complete graph: every token attends to every other token, so the "graph" connecting the tokens in a sequence has an edge between every pair. Attention weights are, in this reading, learned edge weights recomputed fresh for every input. Knowledge graphs take the opposite approach: vertices are real-world entities, edges are explicit relations ("Paris" —capital-of→ "France"), and models are trained to predict missing edges — the graph *is* the knowledge representation, not just a computational scaffold. Finally, PageRank, Google's original ranking algorithm, treats the entire web as a directed graph and computes each page's importance as a function of a random walk over that graph — closely related to the "steady-state" behavior at the heart of Markov chains.

⬡ AI Connection

The throughline of this whole chapter: BFS's wave expansion is GNN message passing; DAG topological order is exactly how backpropagation sequences its computation; MSTs and trees underlie decision trees and clustering; shortest-path relaxation resembles value iteration in reinforcement learning; and bipartite matching shows up in routing tokens to experts. Graph theory isn't a prerequisite you check off before AI — it's the vocabulary AI architectures are already written in.
